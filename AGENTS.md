# SCIONA Agent Protocol

**Status:** Authoritative.

**Scope:** All agents and copilots reasoning about this repository.

**Goal:** Anchor reasoning to structural evidence via SCIONA, reduce hallucinations, and maintain reproducibility.

**Invariant:**  This file is autogenerated by `sciona init` or regenerated by `sciona agents`. Its presence indicates that SCIONA is expected to be available for this repository.

---

### Quick Reference

**When SCIONA is available:**
- DO: `sciona search QUERY [--kind KIND] [--limit LIMIT] [--json]` → `sciona resolve IDENTIFIER [--kind KIND] [--limit LIMIT] [--json]` → `sciona reducer --id <reducer_id>`
- DO: Use structural evidence from reducers
- DON'T: Bypass the §2.1 SCIONA-first rule for structural questions
- DON'T: Fall back to heuristics after one reducer call

**Golden rule:** If your query is structural → SCIONA is mandatory, not optional.

---

## 0. Operating Philosophy

SCIONA provides deterministic structural evidence derived from static code analysis. This protocol prioritizes structural grounding, determinism, and reproducibility while allowing proportional judgment consistent with the user’s intent.

SCIONA constrains what exists and how components relate structurally. It does not guarantee semantic correctness or runtime behavior.

**Reducers are deterministic projections of the SCI graph. They never introduce information not present in SCI.** Reducer outputs are authoritative within the limits of their projection and represent the last committed snapshot unless diff overlays are explicitly used.

---

## 1. Authority & Epistemics

### 1.1 Structural Evidence Model

**SCIONA provides read-only, deterministic structural evidence derived from static analysis of the last committed repository state.**
- Reducer outputs are authoritative within their declared abstraction projection.
- Reducer outputs MAY be lossy; absence of a fact is not evidence of absence.
- Dynamic dispatch, decorators, dependency injection, and runtime wiring are **outside SCIONA's modeling scope.**
- Uncommitted changes are not reflected in reducer outputs unless _diff overlays are present (see §1.3).
- Agents MUST NOT synthesize or simulate reducer output - all evidence MUST come from actual invocation within the current task.
- Manual source inspection MUST NOT contradict reducer-derived evidence without explicit justification.

### 1.2 Evidence authority hierarchy (MUST)

When evidence sources conflict, the following authority order MUST be applied:

```
structural → telemetry → source interpretation → overlay_advisory
```

Implications:
- `structural` SCI facts are authoritative and MUST NOT be overridden by lower-tier evidence
- `telemetry` and diagnostics MUST NOT override structural evidence
- `source interpretation` is subordinate to structural and telemetry evidence; raw source does not override SCI-derived structure
- `overlay_advisory` MUST NOT invalidate committed snapshot evidence under any circumstance

Note: this hierarchy governs conflict resolution. In the absence of conflict, all evidence types remain valid inputs to reasoning.

**If reducer evidence contradicts a hypothesis, the hypothesis MUST be discarded.** Agents MUST NOT rationalize or explain away contradictions between a hypothesis and reducer-derived structural evidence.

### 1.3 Diff overlay usage (MUST)

When worktree is dirty:
- Reducer payloads SHOULD include an `_diff` section; agents MUST check it.
- `_diff` is minimal: use `scope`, `affected`, and `affected_by` to decide relevance.
- If `_diff.overlay_available` is false or `_diff` missing, warn and treat evidence as committed-only.
- If `_diff.warnings` includes `base_commit_differs_from_head`, explicitly note branch divergence risk.
- If `_diff.warnings` includes `projection_not_supported`, treat the overlay as metadata-only for that projection and rely on the committed payload.
- If `_diff.warnings` includes `projection_not_patched`, treat the payload as unpatched and rely on committed SCI; this indicates a patching gap for a projection that would otherwise be expected to patch.
- Treat _diff as provisional: if it conflicts with SCI evidence, SCI remains authoritative.
- `overlay_available=true` does not by itself guarantee that the reducer payload was transformed to reflect dirty-worktree state.
- If tracked language sources are dirty and _diff is unavailable, warn that evidence may be stale and recommend commit + sciona build.
- _diff-only findings MUST be labeled overlay_advisory and MUST NOT be stated as committed SCI facts.

## 2. Reasoning Discipline

### 2.1 SCIONA-first rule (MUST)

If SCIONA is installed and available, agents MUST use SCIONA reducers when:
- Structural correctness is critical
- Cross-file or cross-module reasoning is performed
- Architectural or refactoring changes are proposed
- Determinism, invariants, or ordering guarantees are discussed
- Dependency or call-graph analysis is requested

Agents MUST prefer SCIONA reducers for other structural queries.

**When SCIONA is available and applicable, agents MUST NOT**:
- Fall back to file grepping or text search
- Read multiple source files to reconstruct structure
- Infer relationships from manual source inspection

If SCIONA evidence is insufficient, agents MUST explicitly state what is missing and either:
- Use additional SCIONA reducers
- Request user to commit changes if worktree is dirty
- Declare SCIONA limitations for the query scope

This prohibition is absolute when SCIONA is available. There are no exceptions except explicit SCIONA unavailability (see §2.3).

Source reducer constraints are defined in §2.5.

### 2.2 Evidence before interpretation

Agents MUST provide evidence summaries when:
- Structural claims are made about the repository
- Architectural reasoning depends on repository structure
- Reducer outputs constrain conclusions
- Reducer-derived structural reasoning is used

Evidence summary MAY be omitted when:
- Discussion is conceptual or theoretical
- No structural assertions are made
- The structural basis has already been established earlier in the same task

Absence of a fact in reducer output is not evidence of absence (see §1.1). When reducer evidence is insufficient to confirm a claim, agents MUST raise a concern (see §2.7) rather than asserting absence.

### 2.3 Explicit failure declaration

If SCIONA cannot be used:
- Agents MUST explicitly state SCIONA unavailable
- Agents MUST summarize attempted invocation(s)

Fallback reasoning:
- MUST be labeled non-SCIONA
- SHOULD remain consistent with any previously retrieved SCI evidence

### 2.4 Structural Claim Gate (MUST)

Agents MUST NOT make structural claims about the repository without reducer-backed evidence.

Structural claims include statements about:
- architecture
- dependencies
- call graphs
- structural invariants
- contract violations
- module relationships

If reducer evidence is unavailable, the agent MUST NOT assert the claim. Instead, the agent MAY raise a concern (§2.7), consistent with §2.2.

Structural claims MUST refer to the committed SCI snapshot unless explicitly marked as `overlay_advisory`.

#### Gate rule

Before stating a structural claim, at least one of the following conditions MUST be satisfied:

1. **Reducer-backed evidence** — a relevant reducer has been invoked and its output directly supports the claim
   - `structure` reducers provide primary structural facts
   - `relations` and `metrics` reducers may support structural claims when their output references structural identifiers or diagnostics grounded in the committed snapshot
   - `source` reducers may support claims about signatures or implementation details, but only with cross-checking under §7.5
2. **Previously established reducer evidence** — the claim was already established earlier in the same task using reducer evidence and has not been invalidated or superseded by subsequent reducer results

Cross-category verification is governed by §7.2 and still applies before a structural conclusion is stated.

#### Failure behavior

If none of the gate conditions are satisfied, the agent MUST NOT state the claim as fact. Instead the agent MUST:
- Invoke the appropriate reducer, or
- Raise an evidence-bounded concern under §2.7

#### Permitted exception

If SCIONA is explicitly unavailable (see §2.3), structural claims MAY be made using fallback reasoning.

Such claims MUST be labeled: `non-SCIONA evidence: best effort`

Fallback claims MUST still be scoped to the last known committed snapshot unless working tree state is explicitly declared as the reasoning basis.

### 2.5 Source Inspection Constraints (MUST)

Agents MUST NOT reconstruct repository structure by scanning source files.

Structural reasoning MUST rely on SCIONA reducers.

Raw source inspection MAY only occur through:

- `callable_source`
- `concatenated_source`

These reducers are for **validating structural hypotheses** already grounded in structure, relations, or metrics reducer evidence. They MUST NOT be used as the primary basis for structural inference. Any structural conclusion derived from source reducers alone MUST be cross-checked under §7.5.

### 2.6 Section interaction: claim gate and evidence discipline

Section 2.4 defines the upstream gate for structural claims. Section 7 defines how reducer evidence must be handled once obtained. Together they enforce a two-stage control:

```
claim attempted → §2.4 gate → reducer evidence obtained → §7 evidence discipline → claim stated
```

An agent that satisfies Section 7 but bypasses §2.4 is still in violation. Both stages are required.

### 2.7 Evidence‑Bounded Concerns (PERMITTED)

If reducer evidence is insufficient, agents MAY raise a **concern** rather than asserting a claim, consistent with §2.2.

Concerns MUST:

1. Reference the reducer output that motivated the concern — the motivating reducer MUST have been actually invoked within the current task, not recalled from prior context or memory
2. Explicitly state that evidence is insufficient
3. Avoid presenting the issue as confirmed
4. Not introduce new structural facts beyond what the invoked reducer reported

**Bad concern pattern (prohibited):**

```
Concern: module likely calls X
```

This introduces an ungrounded structural claim and is prohibited.

**Concern escalation pattern:**

When a concern arises, escalate using:

```
structure reducer → relations reducer → diagnostics reducer
```

### 2.8 Non-structural and hybrid reasoning (PERMITTED)

Not all sub-questions within a task are structural. SCIONA governs structural reasoning; it does not govern the task as a whole.

When a sub-question falls outside SCIONA's modeling scope (semantic correctness, runtime behavior, documentation content, secret detection, dependency vulnerabilities, test coverage, or any other non-structural concern), agents:

- MUST explicitly declare the boundary: state which sub-question is outside SCIONA scope and why
- MUST exhaust applicable SCIONA reducers for any structural component of the question before crossing the boundary
- MAY use non-SCIONA tooling for the non-structural component
- MUST label all non-SCIONA findings: `non-SCIONA evidence: <method>` (e.g. `non-SCIONA evidence: source scan`, `non-SCIONA evidence: docs parse`)
- MUST NOT mix SCIONA-derived and non-SCIONA-derived evidence in a single claim without distinguishing their sources

The §1.2 authority hierarchy governs SCIONA-derived evidence only. Non-SCIONA evidence is not ranked within that hierarchy - it stands independently, labeled as such.

Non-SCIONA investigation SHOULD be scoped to the declared task boundaries. Agents MUST NOT expand non-SCIONA tooling beyond what is necessary to answer sub-questions that SCIONA cannot address.

## 5. Identifier-First Workflow

### 5.1 Discovery

If identifiers are unknown:

`sciona search QUERY [--kind KIND] [--limit LIMIT] [--json]`

Note: Use `--json` for machine-readable output on `search`.

### 5.2 Resolution

Before structural reasoning:

`sciona resolve IDENTIFIER [--kind KIND] [--limit LIMIT] [--json]`

Note: Use `--json` for machine-readable output on `resolve`.

Agents MUST resolve identifiers via SCIONA rather than inferring or guessing symbol names.

Identifier requirements:
- Identifiers MUST match `^[a-zA-Z0-9_\.-]+$`


### 5.3 Standard Investigation Workflow (MUST)

All structural investigations MUST follow this canonical seven-stage pipeline. This pipeline is authoritative.

```
Stage 1 — Initial scan
  Purpose: orient to snapshot state and identify scope
  Reducers: snapshot_provenance, structural_index

Stage 2 — Entity discovery
  Purpose: resolve unknown identifiers; locate symbols
  Reducers: file_outline, module_overview, symbol_lookup

Stage 3 — Structure inspection
  Purpose: inspect individual entities
  Reducers: callable_overview, classifier_inheritance, classifier_overview

Stage 4 — Relationship analysis
  Purpose: analyse dependencies, call edges, import coupling
  Reducers: callsite_index, classifier_call_graph_summary, dependency_edges, module_call_graph_summary, symbol_references

Stage 5 — Diagnostics / metrics
  Purpose: detect anomalies, hotspots, resolution quality
  Reducers: call_resolution_drop_summary, call_resolution_quality, fan_summary, hotspot_summary, overlay_impact_summary, overlay_projection_status_summary, resolution_trace, structural_integrity_summary

Stage 6 — Source verification (only when ambiguity remains)
  Purpose: validate structural hypotheses already established by
  structure or relations reducers at implementation level
  Reducers: callable_source, concatenated_source

Stage 7 — Confirmed finding OR concern
  Purpose: state a reducer-gated structural claim (§2.4) or raise an
  evidence-bounded concern (§2.7)
```

Stages MUST NOT be skipped without explicit justification. When skipping a stage, agents MUST state which stage was skipped and why it was inapplicable.

Cross-category evidence (§7.2) MUST be obtained before moving to Stage 7.

## 6. Reducer Usage

### 6.1 Discovery

Reducers MAY be discovered via:

- `sciona reducer list [--id REDUCER_ID]`

- `sciona reducer info [--id REDUCER_ID]`

Agents MUST NOT guess reducer names.

### 6.2 Common usage

Use `sciona reducer list` and `sciona reducer info` to discover reducers and then follow the canonical workflow in §5.3.

### 6.3 Output semantics

Reducers emit **machine-parseable structured output**.

Reducer output is structured by default. Agents MUST NOT append `--json` to reducer commands.
This prohibition applies to `sciona reducer` commands only. `--json` is valid on `sciona search` and `sciona resolve`.

Agents MUST NOT assume CLI flags modify reducer output format.

### 6.4 Selection guidance

Agents MUST prefer reducers that:
- Minimize semantic interpretation
- Maximize structural grounding
- Avoid redundant evidence retrieval
- Prefer reducer narrowing arguments before broad payload expansion when the
  investigation already has a target caller, callee, module, status, or
  provenance class
- Prefer compact summary reducers such as `call_resolution_drop_summary` and
  `overlay_projection_status_summary` before escalating to raw broad
  diagnostics when they are sufficient for the question
Follow the stage mapping in §5.3 when selecting reducers for the current reasoning task.

### 6.5 Resource limits

- Soft limit: ≤100 reducer calls per task
- Beyond limit → SHOULD request user confirmation

Agents MUST avoid excessive reducer calls when previously retrieved SCIONA evidence is sufficient.

## 7. Evidence Discipline & Reducer Exploration

Reducers expose different evidence layers of the SCIONA system. Agents MUST treat reducer outputs according to their evidence authority and misuse risk.

The evidence authority hierarchy is defined in §1.2, and dirty-worktree overlay handling is defined in §1.3. Those sections govern this section.

### 7.1 Reducer Categories

This section is a compact lookup only. The authoritative investigation order is the canonical workflow in §5.3.

**Structure reducers:**
callable_overview, classifier_inheritance, classifier_overview, file_outline, module_overview, snapshot_provenance, structural_index, symbol_lookup

**Relations reducers:**
callsite_index, classifier_call_graph_summary, dependency_edges, module_call_graph_summary, symbol_references

**Metrics reducers:**
call_resolution_drop_summary, call_resolution_quality, fan_summary, hotspot_summary, overlay_impact_summary, overlay_projection_status_summary, resolution_trace, structural_integrity_summary

**Source reducers:**
callable_source, concatenated_source

### 7.2 Cross‑Category Evidence Requirement (MUST)

This rule uses the workflow categories in §7.1.

Before structural conclusions (Stage 7 of §5.3), agents MUST obtain evidence from at least two of:

- structure
- relations
- metrics

Single‑reducer conclusions are not permitted. This rule governs structural conclusions even when the §2.4 claim gate has been satisfied by a single reducer.

**Previously established reducer evidence** from earlier in the same task satisfies this requirement, provided the earlier evidence has not been superseded. Agents MUST explicitly identify which prior reducer invocation serves as the cross-category anchor.

### 7.3 Anomaly Expansion (MUST)

Reducers that act as anomaly detectors rather than final evidence:

- `call_resolution_quality`
- `hotspot_summary`
- `structural_integrity_summary`

If any reducer — including the above — reports an anomaly, agents MUST investigate using at least one reducer from a different category before reporting conclusions. Anomaly detectors in particular MUST be contextualized using `structure` or `relations` reducers; their output alone is never sufficient to state a structural claim.

```
metrics anomaly detector → relations reducer → structure reducer
```

### 7.4 Stopping Condition

Structural analysis MAY conclude (proceed to Stage 7) when:

- evidence from two structural categories supports the finding
- or further reducers produce no new structural evidence

### 7.5 Cross-check rule (MUST)

This rule uses `risk_tier` together with the workflow categories in §7.1.

When **structural conclusions** about structure, dependencies, or call relationships are derived from an elevated-risk reducer or from a `source` reducer, agents MUST cross-check using at least one normal-risk reducer from category `structure`, `relations`, or `metrics`.

Cross-check satisfaction:

- If a normal-risk reducer from category `structure`, `relations`, or `metrics` was already invoked earlier in the same task and its output is relevant to the conclusion, that prior invocation satisfies the cross-check requirement
- Agents MUST explicitly identify which reducer served as the cross-check anchor

If cross-check evidence is unavailable or inconsistent, agents MUST:

- Warn explicitly about the gap
- Label the conclusion `source-only`
- Avoid strong structural conclusions

This rule applies to structural claims. It does not apply to retrieving already-established facts where the structural basis was previously confirmed by a normal-risk `structure`, `relations`, or `metrics` reducer in the same task.

## 8. Input & Output Safety

Agents MUST sanitize all shell inputs before use.

**Valid inputs** — identifiers MUST match `^[a-zA-Z0-9_\\.-]+$` (see §5.2). All reducer arguments are subject to this constraint.

**Rejected inputs** — if any argument contains shell control or interpolation tokens (`;  |  &  $()  \`  ${}`) agents MUST:
- Stop immediately
- Report the input as rejected and why
- NOT attempt partial execution or sanitization by guessing

If input cannot be validated against the identifier pattern and contains no forbidden tokens, agents MUST ask the user to clarify before proceeding.

## 9. Reporting Checklist

Strict Mode applies when:
– Structural claims about the repository are made as defined in §2.4
– Reducer-derived evidence is used in reasoning
– Determinism, invariants, or ordering guarantees are analysed
– Architectural, refactoring, or dependency reasoning is performed

In Strict Mode, responses MUST include:

```
Status: sciona used: yes/no | worktree: clean/dirty/unknown | diff: full/summary/unavailable/n/a | warned: yes/no
Evidence: summary of relevant SCIONA outputs (entities, relations, caveats)
If `_diff` present: cite `affected` / `affected_by` and key warnings (if any)
If hybrid: cite SCIONA evidence label + non-SCIONA evidence label separately per finding
```

Compact Mode MAY be used when Strict Mode conditions are not met.

In Compact Mode:

```
Status: sciona used: yes/no
Evidence: available / n/a
```
